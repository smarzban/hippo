### MED-01 · [access-control] Admin can revoke an owner's bearer token (mutation above tier)
- **Where:** `src/hippo/api.py:861-867`  ·  verdict: real  ·  confidence: high
- **What:** DELETE /tokens/{token_id} routes any caller with rank>=1 to store.revoke_token_any(token_id), which deletes ANY token by id with no tier check (storage.py:572-575). A rank-1 admin can therefore revoke a rank-2 owner's personal access tokens — a mutation against an account ABOVE the admin's tier (denial-of-service against owner-held CLI/MCP automation). This directly contradicts the tier invariant the rest of the file deliberately enforces: _require_folder_tier (api.py:295-304) blocks admins from mutating owner-tier folders, create_user (api.py:675-677) and admin_reset_password (api.py:753-755) both compare the target's EFFECTIVE role against the caller's and 403 if the target outranks the caller. Token revocation is the one admin mutation that skips this check. The test test_revoke_token_any_ignores_owner (tests/test_storage_tokens.py:20) and test_tokens_cross_user_revoke_blocked_for_user_allowed_for_admin (tests/test_api_settings.py:138) only assert admin-revokes-peer, never admin-vs-owner, so the gap is untested.
- **Fix:** In delete_token, before revoking another user's token, resolve the token owner's effective role and 403 if it outranks the caller — mirroring admin_reset_password. E.g. look up the token's owner email (add a storage helper like token_owner(token_id)->email|None), compute target_role = 'owner' if owner in settings.admin_email_list else stored role, and raise 403 if rank(target_role) > rank(user.role). Keep self-revoke (revoke_token matching user_id) unconditional. Add a regression test asserting an admin gets 403 (or 404) revoking an owner-owned token.
- **Verifier:** Verified against the actual code. api.py:861-867 (delete_token): any caller with rank(user.role) >= 1 — which includes a rank-1 admin, not just owners — is routed to store.revoke_token_any(token_id). storage.py:572-575 confirms revoke_token_any deletes ANY token by id with no tier or ownership check. So a rank-1 admin can revoke a rank-2 owner's bearer tokens. The cited inconsistency is real and the file itself documents the opposing invariant: _require_folder_tier (api.py:295-304) blocks admins from managing owner-tier folders; create_user (api.py:675-677) and set_user_role (api.py:712-714) compare against caller tier; admin_reset_password (api.py:749-755) computes target_role = 'owner' if target in settings.admin_email_list else creds['role'] and 403s if rank(target_role) > rank(user.role) precisely to stop a rank-1 admin from hijacking a bootstrap owner's credential. Token revocation is the one admin mutation that skips this effective-role check. Test gap confirmed: tests/test_storage_tokens.py:20 (test_revoke_token_any_ignores_owner) exercises a single user. tests/test_api_settings.py:138 has boss@x.com (owner-tier via admin_emails='boss@x.com') revoke its OWN token (created at line 142 with the admin header, deleted at line 146 with the same admin header) — it proves owner-revokes-own and user-blocked, never admin-vs-owner. No test covers a genuine rank-1 admin revoking a higher-tier owner's token. Proposed fix is feasible: list_all_tokens() already JOINs tokens to user emails, so a token_owner(token_id)->email|None helper is straightforward, and the effective-role guard can mirror admin_reset_password exactly. Severity calibration: impact is denial-of-service against owner-held CLI/MCP automation, not privilege escalation — a revoked token can simply be recreated by the owner, and minting a token already carries the caller's own role (no escalation via this path). It is a genuine, exploitable tier-invariant violation that directly contradicts an explicitly-documented rule applied everywhere else in the file, but bounded to availability. MEDIUM is appropriate.

### MED-06 · [concurrency] reindex() vs concurrent upsert_document() race strands chunks with no vector
- **Where:** `/home/patch/TGC-PRJ/hippo/src/hippo/storage.py:701-734`  ·  verdict: real  ·  confidence: high
- **What:** reindex() snapshots all chunk rows under the lock (line 709: `SELECT id, text FROM chunks`), RELEASES the lock to embed outside it (line 713), then re-acquires the lock to DROP/recreate chunk_vec and repopulate ONLY from the stale snapshot (lines 721-728). If an upsert_document (or sync_folder/ingest) commits new chunk rows during the embedding window, those new chunk ids are absent from reindex's snapshot. After the atomic DROP+recreate, the new chunks exist in `chunks` (and chunks_fts via the trigger) but have NO row in the rebuilt chunk_vec, so they are permanently invisible to vector KNN (_search_vec) until the next full reindex — a silent retrieval gap, not an error. Both methods serialize on the same lock individually, but reindex's read-embed-write is not one critical section, so interleaving is possible. Risk is mainly within a single server process (or any second writer process under WAL); reindex is documented as a CLI op but nothing enforces that no ingest runs concurrently.
- **Fix:** Either (a) re-read the current chunk set inside the final lock and embed any ids added since the snapshot before the swap (loop until the snapshot is current), or (b) take the snapshot, embed, then inside the final lock verify `SELECT max(id), count(*) FROM chunks` matches the snapshot and abort with a clear 'documents changed during reindex — retry' error if not, or (c) document and enforce that reindex requires exclusive access (no concurrent ingest), e.g. an advisory in-process flag that ingest checks.
- **Verifier:** VERIFIED REAL. The cited code at /home/patch/TGC-PRJ/hippo/src/hippo/storage.py:701-734 matches the finding exactly: 1. Line 708-709: `reindex()` snapshots `SELECT id, text FROM chunks` inside `with self._lock`, then releases the lock. 2. Lines 710-720: the network embed loop runs with NO lock held. 3. Lines 721-728: it re-acquires the lock, does `DROP TABLE IF EXISTS chunk_vec`, recreates the vec0 table, and repopulates via `executemany(INSERT ... )` using `new_vectors` — which is derived ONLY from the stale snapshot (`rows`). The read-embed-write is therefore three separate critical sections, not one. `upsert_document` (lines 125-150, its own `with self._lock, self.con` block) inserts both a `chunks` row and its matching `chunk_vec` row. If an upsert/sync/ingest commits during reindex's embedding window, that new chunk: (a) is absent from reindex's snapshot, and (b) its freshly-inserted chunk_vec row is destroyed by reindex's `DROP TABLE`. After the swap the chunk exists in `chunks` (and chunks_fts via trigger) but has no chunk_vec row, so `_search_vec` (line 662-668, KNN over chunk_vec) cannot return it — a silent, permanent-until-next-reindex vector-retrieval gap. (It is still reachable via FTS/grep, so the gap is partial, not total invisibility — the finding correctly notes this.) The `_lock` is a per-Storage `threading.Lock` over one shared connection (line 79), so the in-process race is real; nothing in cli.py (reindex at line 171-179) or api.py enforces that no ingest runs concurrently, and across processes (a running `hippo serve` + a `hippo reindex` CLI against the same DB under WAL) the locks are entirely separate. The finding does NOT misread the code and the risk is not mitigated elsewhere. Severity MEDIUM is appropriate: genuine silent data-integrity defect, but the windows only overlap during a deliberate maintenance op (reindex) concurrent with active ingest, which is uncommon; impact is degraded recall, not a crash or access-control leak. Proposed fixes (re-embed the delta inside the final lock, detect-and-abort on snapshot mismatch, or enforce exclusive access) are all sound; the abort-on-mismatch or re-embed-delta options are the most robust.
